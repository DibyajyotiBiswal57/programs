# FAQ

Common questions about working with the Programming Solutions Universe repository.

## General

**How do I choose the correct filename?**  
Use the filename listed in `questions.md` for the problem you are solving. Keep the four-digit prefix and snake_case name exactly as shown, and append the language extension.

**Where can I see current progress?**  
Check `status.md` for the source of truth. The live status page at https://dibyajyotibiswal57.github.io/programs/ is generated from it.

**Can I edit `index.html` directly?**  
No. `index.html` is generated by `generate_html.py`. Update `questions.md` or `status.md` instead and let automation regenerate the page.

**What languages are supported?**  
There are 17+ language folders (Python, Java, C, C++, Go, Ruby, etc.). Add new languages through the issue template for new language support.

## Working on solutions

**Do I need to add a comment in new solution files?**  
Yes. Include a short header comment that states the problem or its goal, following the style used in that language folder.

**How should I format code?**  
Use the conventions already present. Java is formatted with google-java-format (2-space indents), Python uses 4-space indents, and other languages follow their folder’s style. Run language formatters if available.

**How do I run a quick test?**  
Execute the file with the appropriate runtime (`python3 python/<file>.py`, `go run go/<file>.go`, `gcc c/<file>.c -o /tmp/a.out && /tmp/a.out`, etc.). There is no centralized test harness for the challenge programs.

**What if I see unrelated lint failures?**  
Focus only on the files you changed. Do not modify unrelated code unless necessary for your change.

## Contributing and reviews

**Do I need to update documentation when adding solutions?**  
Update `status.md` when adding new implementations. If you change process or guidelines, update the relevant docs or wiki page.

**How do I propose a new problem or language?**  
Open an issue using the provided templates (`New Problem` or `New Language Implementation`) and share sample code or rationale as requested.

**What is the preferred workflow for PRs?**  
Keep changes minimal, describe the problem solved, mention any manual testing, and avoid touching generated assets. Use the canonical filenames and run available linters for your language.

**Who maintains the repository?**  
Ownership is defined in `.github/CODEOWNERS`. Reviewers are assigned automatically based on file paths.

## Troubleshooting

**My program cannot read input correctly. What should I do?**  
Follow the input/output examples in `questions.md` when provided. Normalize whitespace and ensure your language’s standard input handling matches the problem description.

**The status page did not update after my change.**  
Ensure `status.md` was updated and committed. The GitHub Action regenerates `index.html` on push; if running locally, execute `python3 generate_html.py` and commit only the source files, not the generated HTML unless automation adds it.

**I am unsure which branch to target.**  
Use the default branch unless a maintainer specifies otherwise. Keep PRs focused on a single problem or documentation improvement.
